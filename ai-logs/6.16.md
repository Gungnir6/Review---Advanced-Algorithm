# 1. Section III 主要讲了什么

Section III 主要介绍并分析论文的基础结构：**rotated radix trie（旋转基数树）**。

这一节的目的不是直接给出最终的超低失败概率结构，而是先构造一个可以替代传统哈希表基本功能的数据结构。它和普通哈希表一样，希望实现线性空间和高概率常数时间操作；但不同的是，它不依赖哈希函数把 key 映射到桶中，而是把随机性放进 trie 的节点数组结构里。

可以把 Section III 看作整篇论文的技术基础：

- 后面的 **amplified rotated trie** 是在它的基础上降低失败概率；
- 后面的 **budget rotated trie** 是在它的基础上减少随机比特；
- 所以如果不理解 rotated radix trie，后面的两个结构会很难看懂。

# 2. rotated radix trie 和普通 radix trie 有什么区别

普通 radix trie 会根据 key 的不同片段逐层向下走。每一层节点通常对应 key 的一段 bit，节点的孩子位置由这一段 bit 的取值决定。

rotated radix trie 的主要区别在于：每个内部节点的孩子数组并不是按原来的顺序直接使用，而是会加上一个随机的循环偏移量。

也就是说，如果某个节点原本应该根据 child index $c$ 进入某个位置，那么 rotated radix trie 会用类似下面的方式重新映射：

$$
\phi(s,c) = (r_s + c) \bmod n
$$

其中：

- $s$ 表示当前的源节点；
- $c$ 表示孩子编号；
- $r_s$ 是该节点对应的随机旋转量。

这个随机旋转并不是传统意义上的哈希函数。它没有为每个 key 单独计算一个哈希值，而是对节点中的稀疏数组整体做旋转。

# 3. 为什么说它没有直接使用哈希函数

传统哈希表的核心是一个哈希函数 $h(x)$，它直接决定 key $x$ 应该放到哪个桶中。失败概率很大程度上取决于这个哈希函数族是否足够随机、是否能高效求值。

rotated radix trie 的思路不一样。它仍然利用随机性，但随机性不是以 $h(x)$ 的形式出现，而是以节点数组的随机旋转量出现。

换句话说，key 的路径仍然来自 trie 的结构；随机性只是在每个节点处改变数组位置的排列方式。这样就绕开了“必须构造一个强哈希函数族”的限制。

因此论文说它是一个 “hash table without hash functions”，不是说完全没有随机性，而是说随机性没有通过传统哈希函数来使用。

# 4. 随机旋转为什么有助于控制负载

可以把每个非空孩子指针看作一个 ball，把数组位置看作 bin。普通 trie 中，如果很多节点的孩子位置集中到相同下标，就可能造成某些位置负载过高。

rotated radix trie 给每个源节点一个随机偏移 $r_s$。这样同一个节点中的孩子位置会整体平移到随机位置上，不同源节点之间的冲突会被打散。

直观理解是：

- 不同节点的数组旋转量彼此随机；
- 原本可能集中到同一位置的孩子指针，会因为不同旋转量被分散；
- 只要每个 bin 中的元素数量保持在 $polylog(n)$ 以内，就可以用动态 fusion node 等小规模字典结构在常数时间内处理。

因此 rotated radix trie 的分析重点就是证明最大负载不会太大，从而保证每次操作只需要处理很小的局部结构。

# 5. Section III 和后面两个结构的关系

Section III 给出的 rotated radix trie 本身已经能说明：不直接使用哈希函数也可以构造类似哈希表的随机化字典。

但是它还不是论文最终结果。它的作用更像一个基础版本：

- Section IV 在此基础上构造 **amplified rotated trie**，目标是把失败概率进一步降低到接近最优；
- Section V 在此基础上构造 **budget rotated trie**，目标是只使用很少的随机比特；
- 后续的 succinct dictionary 结果也依赖这些结构。

所以理解 Section III 时，重点不是马上完全看懂所有概率证明，而是先理解结构本身：节点如何旋转、元素如何映射、负载为什么会被随机化。

# 6. Section III 后面还需要重点看什么

后面继续阅读 Section III 时，应该重点关注 rotated radix trie 的证明部分，而不只是停留在结构描述上。

比较重要的几个问题包括：

- 论文如何形式化定义每个 bin 的 load；
- 为什么随机旋转之后最大负载可以被控制在 $polylog(n)$；
- rotated radix trie 如何保证总空间仍然是线性的；
- 插入、查询和删除操作为什么可以在高概率下保持常数时间；
- 后续 amplified rotated trie 是如何在这个基础上进一步降低失败概率的。

也就是说，Section III 的重点不只是“随机旋转”这个想法本身，还包括它背后的负载分析。只有把这一部分弄清楚，后面 Section IV 和 Section V 的改进才比较容易理解。
